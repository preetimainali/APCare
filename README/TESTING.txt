1.  TESTING OVERVIEW
================================================================================

Testing approach: Manual testing (appropriate for a 24-hour hackathon prototype).

The application has no automated test suite. All testing was performed manually
using the procedures documented below. Each test case includes:

  • ID          — unique identifier
  • Area        — feature being tested
  • Steps       — how to perform the test
  • Expected    — what should happen
  • Status      — PASS / FAIL / NOT TESTED

For model validation, the application includes a dedicated Validation page that
performs multi-window walk-forward backtesting with quantitative accuracy metrics.

5.  PREDICTIVE MODEL VALIDATION
================================================================================

The prediction model is validated quantitatively through multi-window
walk-forward backtesting, visible on the /validation page.

TEST-PRED-01: Backtest Windows
  Approach: Train on historical data, predict unseen future, compare
  Windows:
    Window 1: Train Jan 1–20  -> Predict Jan 21–27
    Window 2: Train Jan 1–25  -> Predict Jan 26–Feb 1
    Window 3: Train Jan 1–30  -> Predict Jan 31–Feb 6
  Verification:
    • Each window uses only past data for training (no data leakage)
    • Predictions are generated for a full 7-day horizon
    • Actuals are compared against predictions

TEST-PRED-02: Accuracy Metrics Verification
  Metrics checked per window:
    • MAE  — Mean Absolute Error (patients)
    • RMSE — Root Mean Squared Error (patients)
    • MAPE — Mean Absolute Percentage Error (%)
    • Bias — Average signed error (positive = over-prediction)
    • Coverage — % of actuals within confidence bands
  Expected:
    • MAPE should be under 15% for most facilities (Fair or better)
    • Coverage should be above 80% for meaningful confidence bands
    • Bias should be small and not consistently positive or negative

TEST-PRED-03: Trust Level Derivation
  Rules:
    • HIGH trust:      average MAPE <= 5%
    • MODERATE trust:  average MAPE <= 12%
    • LOW trust:       average MAPE > 12%
  Verification:
    • Trust badges on validation page match the computed MAPE ranges
    • Trust level carries through to alert confidence tags

TEST-PRED-04: Risk Classification
  Rules:
    • CRITICAL — forecast peak >= 95% of capacity
    • HIGH     — forecast peak >= 90% of capacity
    • MODERATE — forecast peak >= 80% of capacity
    • LOW      — forecast peak < 80% of capacity
  Verification:
    • Risk tiers on validation page match peak forecast / capacity ratios
    • Classifications are consistent across windows

  Steps:
    1. Click gear icon in alert panel header

    3. Adjust ICU critical threshold to 80%

  Expected:
    • Settings modal opens with sliders

  Steps:

    2. Save and return to dashboard
  Expected:
    • Toast notifications appear at bottom-right
    • Slide-in animation
    • Auto-dismiss after 8 seconds
    • Manual dismiss via X button

TEST-ALERT-05: Alert Deduplication
  Steps:
    1. Observe alerts for a facility with both warning and critical issues
  Expected:
    • Info alerts suppressed when warning/critical exists for same category
    • Suppressed count shown as summary note

TEST-ALERT-06: Forecast-Based Alerts
  Steps:
    1. Set forecast risk thresholds low enough to trigger alerts
    2. Observe forecast alerts in the panel
  Expected:
    • Forecast badge appears on prediction-based alerts
    • Trust level badge (HIGH/MODERATE/LOW) shows confidence

  • No CI/CD pipeline for automated test execution
