1.  PROJECT OVERVIEW
================================================================================

"Hospital Command Center" is a hackathon prototype built for HCA Healthcare.
It is a responsive web application that:

  • Visualizes real hospital census data from 10 HCA facilities
  • Shows overall occupancy and ICU capacity in real time
  • Displays historical trends (census, ICU, admissions, discharges, births)
  • Predicts future occupancy using a transparent statistical model
  • Generates configurable alerts when thresholds are exceeded
  • Validates prediction accuracy with multi-window backtesting
  • Supports role-based access control (Admin, CNO, CFO, Analyst, custom)
  • Allows customization by facility and date range
  • Exports professional PDF/CSV executive briefs
  • Simulates secure enterprise sharing (@hca.org restriction)

The prototype uses pre-processed static JSON files instead of a live backend API.
All data originates from a real Excel dataset: "HCA Census Metrics.xlsx".

4.  DATA PIPELINE — RAW EXCEL → DASHBOARD
================================================================================

Source: data/HCA Census Metrics.xlsx
  • 10 HCA hospitals
  • Hourly readings from January 1, 2026 to February 6, 2026
  • Metrics: total_census, icu_occupancy, admissions, discharges, births

Processing (Python scripts, run offline):

  Step 1: INGESTION
    • openpyxl reads every row from the Excel workbook
    • Each row has: facility, timestamp, metric, value
    • Timestamps normalized to ISO 8601 UTC

  Step 2: FACILITY EXTRACTION → facilities.json
    • Unique facility IDs, names, total bed counts, ICU bed counts
    • Result: array of 10 facility objects

  Step 3: SNAPSHOT EXTRACTION → latest-snapshot.json
    • For each facility, the most recent observation of each metric
    • Keyed by facility ID

  Step 4: TREND AGGREGATION → trends.json
    • Full time series per facility per metric
    • Array of {time, value} objects sorted chronologically

  Step 5: PREDICTION GENERATION → predictions.json
    • Transparent statistical model applied per facility:
      ─ 24-hour Exponential Moving Average (EMA) for current baseline
      ─ 72-hour OLS linear regression for trend direction
      ─ Day-of-week × hour-of-day seasonal pattern
      ─ Confidence bands widening with forecast horizon
    • 7-day (168-hour) forecast generated
    • Output: predicted values, upper/lower bounds, driver contributions

  Step 6: VALIDATION GENERATION → validation.json
    • Multi-window backtesting:
      Window 1: Train Jan 1–20 → predict Jan 21–27
      Window 2: Train Jan 1–25 → predict Jan 26–Feb 1
      Window 3: Train Jan 1–30 → predict Jan 31–Feb 6
    • Per-window metrics: MAE, RMSE, MAPE, Bias, Coverage
    • Cross-window averages and trust level derivation
    • Risk classification based on forecast peak vs capacity

  Data Quality Handling:
    • Missing values: forward-filled from previous observation
    • Duplicates: last value kept, earlier duplicates discarded
    • Anomalies: values outside 3 standard deviations flagged
    • Out-of-range: negative values and values > 2× capacity capped

5.  FRONTEND ARCHITECTURE
================================================================================

Framework: Next.js 14 App Router

Route structure:
  /                  → Dashboard (app/page.tsx)
  /validation        → Validation page (app/validation/page.tsx)

Root layout (app/layout.tsx):
  • Loads JetBrains Mono font as CSS variable
  • Wraps all routes in <RoleProvider> for RBAC
  • Renders <AdminPanel> globally (slide-in role management)
  • Sets HTML metadata (title, description)

State management:
  • React Context for roles (RoleProvider)
  • localStorage for persistence (roles, alert config)
  • Component-level useState for UI state
  • useMemo for derived data (snapshots, alerts, timeline)

Data flow:
  Static JSON → lib/data.ts → React components
  No API calls — all data is imported at build time

10. PREDICTION MODEL
================================================================================

Name: Transparent Statistical Forecast
Type: Additive decomposition (no black-box ML)

Components:
  1. 24-hour Exponential Moving Average (EMA)
     • Weights recent observations more heavily
     • Provides the current baseline level
     • Typically contributes ~55–65% of the forecast

  2. 72-hour Ordinary Least Squares (OLS) Linear Trend
     • Fits a line through the last 72 hours of data
     • Captures directional momentum (rising/falling/stable)
     • Applies exponential damping to prevent runaway extrapolation
     • Typically contributes ~15–25% of the forecast

  3. Day-of-Week × Hour-of-Day Seasonal Pattern
     • Calculates average deviations for each combination
     • Captures recurring rhythms (e.g., higher admissions on weekdays)
     • Typically contributes ~15–25% of the forecast

Forecast formula:
  predicted(t) = EMA_baseline + damped_trend(t) + seasonal_adjustment(t)

Confidence bands:
  • Width increases with forecast horizon
  • Based on historical prediction error standard deviation
  • Upper = predicted + z * σ * sqrt(hours_ahead)
  • Lower = predicted - z * σ * sqrt(hours_ahead)

Forecast horizon: 7 days (168 hours)
Applied separately to: total_census, icu_occupancy


================================================================================
11. MODEL VALIDATION & BACKTESTING
================================================================================

Approach: Multi-window walk-forward backtesting

Windows:
  Window 1: Train Jan 1–20 → Predict Jan 21–27
  Window 2: Train Jan 1–25 → Predict Jan 26–Feb 1
  Window 3: Train Jan 1–30 → Predict Jan 31–Feb 6

Per-window metrics:
  • MAE  — Mean Absolute Error
  • RMSE — Root Mean Squared Error
  • MAPE — Mean Absolute Percentage Error
  • Bias — Average signed error (under/over-prediction)
  • Coverage — % of actuals falling within confidence bands

Cross-window summary:
  • Average of all per-window metrics
  • Driver contribution averages

Trust level derivation:
  • HIGH     — average MAPE ≤ 5%
  • MODERATE — average MAPE ≤ 12%
  • LOW      — average MAPE > 12%

Risk classification:
  • CRITICAL — forecast peak ≥ 95% of capacity
  • HIGH     — forecast peak ≥ 90% of capacity
  • MODERATE — forecast peak ≥ 80% of capacity
  • LOW      — forecast peak < 80% of capacity

13. ALERT SYSTEM (CONFIGURABLE)
================================================================================

Architecture:
  lib/alertConfig.ts    — types, defaults, generation logic
  AlertPanel.tsx        — dashboard alert list
  AlertToast.tsx        — popup notifications
  AlertSettingsPanel.tsx — configuration modal

Default thresholds:
  Overall occupancy: warning 80%, critical 90%
  ICU occupancy:     warning 75%, critical 85%
  Forecast risk:     warning 80%, critical 90%

Alert types:
  1. Current state alerts — triggered by latest snapshot exceeding threshold
  2. Forecast alerts — triggered when 7-day peak forecast exceeds threshold
     (only when current value is still below threshold)

Each alert includes:
  • Severity: critical / warning / info
  • Facility name and ID
  • Human-readable message
  • Timestamp
  • Trust level (for forecast alerts)
  • Reason string explaining exactly why it was triggered
  • Threshold value and actual value

Notification styles (configurable):
  • Dashboard — alerts shown in AlertPanel
  • Popup Toast — slide-in notifications at bottom-right
  • Email (simulated) — requires @hca.org email recipient

Prioritization:
  • Sorted by severity (critical first)
  • Info alerts suppressed when warning/critical exists for same category
  • Per-facility cap (default: 4) prevents one facility from dominating
  • Suppressed alerts counted and shown as summary

Persistence: localStorage key "hcc-alert-config"
